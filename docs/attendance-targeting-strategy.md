# 나이스 출결 칸 타게팅 전략

## 목적

이 문서는 나이스 출결관리 화면에서 학생 목록이 스크롤된 상태여도
원하는 학생의 특정 날짜 칸을 안정적으로 찾아 들어가는 현재 방식을 정리한 문서다.

핵심 목표는 다음 두 가지다.

1. 화면이 위에서 시작하지 않아도 학생을 다시 찾을 것
2. 동적 ID나 고정 XPath 없이도 날짜 칸을 정확히 열 것

## 문제 배경

나이스 출결 그리드는 일반 HTML 테이블처럼 고정된 구조가 아니다.

- 화면에 보이는 학생만 DOM에 올라오는 가상 스크롤 형태에 가깝다.
- 스크롤 위치에 따라 같은 행 위치가 다른 학생을 가리킬 수 있다.
- `pt-ac/002001` 같은 ID나 절대 XPath는 신뢰하기 어렵다.
- 사용자가 스크롤을 내린 상태에서 시작하면, 처음 보이는 학생 집합이 달라진다.

이 때문에 "현재 화면에서 몇 번째 줄" 같은 방식은 쓰지 않는다.

## 현재 전략 요약

현재 자동화는 아래 순서로 학생/날짜 칸을 찾는다.

1. 출결 그리드 자체를 찾는다.
2. 가능하면 그리드를 위쪽으로 리셋한다.
3. 현재 보이는 학생 이름들을 읽는다.
4. 목표 학생이 보이면 해당 행을 잡는다.
5. 안 보이면 세로 스크롤을 조금씩 내리면서 다시 읽는다.
6. 목표 학생 행을 찾으면, 그 행 안에서 목표 날짜 칸을 클릭한다.
7. 행 내부 클릭이 실패하면 날짜/학생명 기반 fallback 셀렉터를 사용한다.

## 학생 찾기 방식

학생 이름은 `cl-grid` 안의 `aria-label*='성명'` 셀을 기준으로 읽는다.

사용하는 이유:

- 눈에 보이는 학생 집합만 빠르게 파악할 수 있다.
- 텍스트만 읽는 방식보다 안정적이다.
- 행 순번이 바뀌어도 이름 자체로 다시 찾을 수 있다.

행 탐색은 다음 우선순위를 쓴다.

1. `aria-label`에 학생명이 들어간 grid row
2. 행 텍스트에 학생명이 포함된 row
3. 마지막 fallback으로 학생명 텍스트

즉, "6번 줄 학생"이 아니라 "김태이 행"을 직접 찾는다.

## 스크롤 안전 처리

스크롤이 내려간 상태에서도 동작하게 하기 위해, 학생 탐색 전에 그리드를 가능한 한 위로 올린다.

구현 포인트:

- 출결 그리드에 마우스를 올린 뒤 휠 스크롤을 위로 반복한다.
- 보이는 학생 목록 시그니처가 더 이상 변하지 않으면 상단 경계로 판단한다.
- 그리드 크기를 측정하지 못하는 경우에는 즉시 죽지 않고 reset 단계를 건너뛴다.
- reset 이후에도 학생이 안 보이면 일정 간격으로 아래로 스크롤하면서 다시 탐색한다.

이 구조 덕분에 사용자가 스크롤을 내려놓고 시작해도 다시 상단부터 찾을 수 있다.

## 날짜 칸 찾기 방식

날짜 칸은 학생 행 내부에서 상대 탐색을 먼저 시도한다.

기본 패턴:

- `${day}일`
- `편집창`

예를 들어 3월 3일이면 행 내부에서 `3일`과 `편집창`이 포함된 셀을 찾는다.

이 방식이 실패하면 fallback 셀렉터를 쓴다.

fallback 기준:

- 셀 `title` 안의 날짜
- 셀 `title` 안의 학생명

즉, 1차는 "학생 행 안의 3일 칸", 2차는 "title 기준으로 김태이의 2026.03.03 칸"이다.

## 왜 이 방식이 안정적인가

- 행 번호 대신 학생 이름을 기준으로 움직인다.
- 절대 XPath 대신 화면 의미가 있는 문자열을 쓴다.
- 스크롤로 DOM이 바뀌어도 다시 현재 보이는 학생 집합을 읽는다.
- 날짜 클릭 실패 시 fallback 경로가 있다.

결과적으로, 화면 배치가 조금 바뀌거나 스크롤 위치가 달라도 복구 가능성이 높다.

## 현재 한계

학생/날짜 칸 진입은 지금 안정적이지만, 팝업 내부 입력은 별도 문제로 다뤄야 한다.

특히:

- `결석 질병` 라디오 선택
- 저장 확인 팝업 처리
- 비고등록 흐름 연결

이 부분은 학생/날짜 칸 타게팅과는 다른 문제이므로 분리해서 다루는 것이 좋다.

## 다음 확장 포인트

비고등록까지 이어갈 때도 같은 원칙을 유지한다.

- 먼저 학생/날짜 칸을 안정적으로 연다.
- 그 다음 팝업 내부 조작만 별도 전략으로 처리한다.
- 그리드 탐색과 팝업 조작을 섞지 않는다.

이렇게 분리해두면, 나중에 팝업 UI가 바뀌어도 학생/날짜 찾기 로직은 그대로 재사용할 수 있다.
