# 나이스 출결 자동화 작업 여정 기록

## 왜 이 문서를 남기는가

이 문서는 단순 구현 설명서가 아니다.

처음 아이디어를 잡은 시점부터, 실제로 `npm start`가 하루짜리 출결과 연속출결을 저장하고 후속 확인 팝업까지 통과하게 되기까지 어떤 과정을 거쳤는지 순서대로 정리한 회고 문서다.

나중에 다시 볼 때 다음 질문에 답할 수 있도록 남긴다.

- 왜 로그인 자동화는 하지 않았는가
- 왜 셀렉터보다 텍스트, 구조, 상태 확인을 더 믿게 되었는가
- 왜 `inspect`, `radio-test`, `confirm-test` 같은 보조 모드를 따로 만들었는가
- 왜 중간에 잘못 저장되는 상황이 있었고, 어떻게 안전하게 바꿨는가

## 1. 출발점: 로그인 자동화는 제외하기로 결정

처음 목표는 나이스 출결처리 자동화였지만, 로그인 과정은 범위에서 제외하기로 했다.

이 판단에는 현실적인 이유가 있었다.

- 인증서
- 업무포털 진입
- 나이스 새 창 열림
- 보안 모듈과 세션 정책

이 구간은 자동화 난이도가 높고 불안정성이 컸다.

그래서 방향을 다음처럼 잡았다.

- 사용자가 전용 브라우저를 연다.
- 사용자가 로그인과 나이스 진입까지 한다.
- 자동화는 그 이후부터만 맡는다.

이 결정이 전체 프로젝트를 훨씬 실용적으로 만들었다.

## 2. 첫 번째 인식: 나이스는 고정 셀렉터만 믿으면 안 된다

초기 화면을 보면서 바로 드러난 문제는 개발자 도구에서 보이는 ID와 XPath가 불안정하다는 점이었다.

예를 들면:

- `pt-ac/002001` 같은 동적 ID
- 화면에 따라 바뀌는 구조
- iframe 가능성
- 절대 XPath 의존성

그래서 초반 전략은 다음처럼 정리됐다.

1. 동적 ID는 1순위로 쓰지 않는다.
2. 화면에 보이는 한글 텍스트를 우선한다.
3. `role`, `aria-label`, `title` 같은 상대적으로 안정된 속성을 같이 본다.
4. 실패 시 후보 셀렉터 배열을 순서대로 시도한다.
5. 클릭 성공 여부는 URL보다 실제 화면 신호로 판단한다.

이 원칙은 끝까지 유지됐다.

## 3. 브라우저 연결 방식 만들기

다음 단계는 이미 로그인된 브라우저 세션에 붙는 구조를 만드는 것이었다.

여기서 한 번 운영적인 문제도 확인했다.

- Edge 기준 스크립트를 먼저 만들었지만 사용자 PC에는 Edge가 없었다.
- Chrome 경로는 있었기 때문에 Chrome 중심으로 전환했다.

이 과정에서 정리된 것:

- `start-chrome-debug.ps1`
- `start-edge-debug.ps1`
- `attach-config.json`
- CDP 연결 기반 브라우저 attach 구조

이 시점부터 자동화는 “브라우저를 새로 띄우는 프로그램”이 아니라 “사용자가 열어둔 세션에 붙는 프로그램”으로 자리를 잡았다.

## 4. inspect 모드가 필요하다는 걸 알게 됨

실제 나이스 화면은 눈으로 보는 것과 DOM 구조가 꽤 달랐다.

그래서 메뉴와 팝업을 추적하려면 사람이 개발자 도구를 보는 것만으로는 한계가 있었다.

그 결과 `inspect` 모드를 만들었다.

`inspect`가 하는 일:

- 현재 나이스 창에 연결
- 프레임 구조 수집
- 텍스트/속성/버튼 후보 수집
- 진단 JSON 저장
- 스크린샷 저장

이 모드 덕분에 “지금 자동화가 뭘 보고 있는가”를 눈으로 확인할 수 있게 됐다.

이건 이후 모든 디버깅의 기반이 되었다.

## 5. 출결 화면까지 직접 들어가서 진단하기

처음 `inspect`를 돌렸을 때는 나이스 메인 대시보드가 잡혔다.

그래서 중요한 전환이 있었다.

- 자동화가 필요한 곳은 메인 페이지가 아니라 `학급담임 > 출결관리 > 출결관리 > 조회 완료` 상태의 화면

이걸 기준으로 다시 `inspect`를 돌렸고, 그때부터 실제 업무화면 구조가 보이기 시작했다.

확인한 핵심:

- 제목: `출결관리`
- 학년/반/월 정보가 화면에 텍스트로 존재
- 조회 결과가 일반 테이블이 아니라 `cl-grid`
- 학생 목록은 일부만 DOM에 올라옴

이 시점이 “이건 가상 스크롤 문제를 풀지 않으면 안 된다”는 사실을 확인한 순간이었다.

## 6. 가장 큰 구조 문제: 가상 스크롤

처음엔 학생 목록을 화면에 보이는 순서로 접근할 수도 있겠다고 생각할 수 있었다.

하지만 실제로 확인해보니:

- 화면에는 `Total 27`명이 보이는데 DOM에는 일부만 올라와 있었다.
- 스크롤을 내리면 같은 위치가 다른 학생을 가리킬 수 있었다.

사용자 기억도 정확했다.

- 스크롤을 안 내리면 정보가 안 뜨는 느낌
- 스크롤 내리면 칸 정보가 섞였던 기억

그래서 전략이 완전히 바뀌었다.

이후 원칙:

1. 보이는 몇 번째 행을 믿지 않는다.
2. 학생 이름으로 현재 보이는 행을 찾는다.
3. 없으면 그리드를 조금씩 스크롤한다.
4. 찾으면 그 행 안에서 날짜 칸을 찾는다.

이 부분이 안정화되면서 “스크롤된 상태여도 학생을 정확히 찾는다”는 수준까지 올라갔다.

## 7. 학생/날짜 칸 진입이 안정화된 시점

한동안은 그리드 탐색이 가장 큰 문제였다.

하지만 결국 다음이 가능해졌다.

- 스크롤이 내려간 상태에서도 시작 가능
- 학생명 기준으로 현재 보이는 행 재탐색 가능
- 날짜 칸 정확히 클릭 가능

사용자가 이 시점에서 “찍는 거는 이제 완벽해”라고 확인해줬고, 그때 별도 문서도 만들었다.

이건 프로젝트의 첫 번째 큰 안정화 구간이었다.

## 8. 두 번째 큰 난관: 팝업 내부 라디오 버튼

학생 날짜 칸 진입이 해결되자, 다음 문제는 `일일출결입력` 팝업이었다.

여기서 중요한 오해도 하나 정리됐다.

초기에는 학생 행의 `조회` 칸에 `/`가 들어가는 쪽이 자동화 대상인 줄 알고 잘못 접근한 적이 있었다.

하지만 실제 하루 전체 결석 입력은 그게 아니었다.

정답은:

- 팝업 상단 `결석` 구역에서 `질병`, `미인정`, `기타`, `출석인정` 중 하나를 찍는 것

즉:

- 학생 행 시간칸 입력과
- 하루 전체 결석 입력

은 완전히 다른 흐름이었다.

이걸 정확히 짚고 나서야 라디오 버튼 문제를 따로 풀 수 있었다.

## 9. radio-test를 만든 이유

전체 흐름에서 자꾸 실패하니, 라디오 클릭만 따로 떼어 검증할 필요가 생겼다.

그래서 `radio-test`를 만들었다.

이 모드는:

- 팝업을 이미 사람이 열어둔 상태에서
- `결석 > 질병`만 눌러본다.

이 보조 모드는 매우 중요했다.

왜냐하면 여기서 다음 사실을 확인할 수 있었기 때문이다.

- 라디오 클릭 로직 자체는 성공할 수 있다.
- 문제는 전체 흐름의 다른 단계와 섞일 때였다.

이후 `radio-test`는 전체 자동화와 독립된 디버깅 도구로 자리를 잡았다.

## 10. 라디오 버튼이 왜 그렇게 어려웠는가

초기에 안 됐던 이유는 단순했다.

- `질병` 텍스트를 눌러도 실제 클릭 타깃이 아니었다.
- `결석`, `지각`, `조퇴`, `결과`가 동시에 비슷한 구조로 있었다.
- custom component라 일반 클릭이 잘 안 먹었다.

결국 여러 시행착오 끝에 최종 방식은 이렇게 정리됐다.

1. 활성 상태의 `일일출결입력` 팝업을 찾는다.
2. 그 안에서 `결석` radiogroup만 찾는다.
3. 그 안의 `질병` radio icon의 실제 좌표를 계산한다.
4. 좌표 클릭을 한다.
5. 클릭 후 `.cl-selected` 상태가 실제로 바뀌었는지 확인한다.

이 방식이 들어가고 나서야 라디오 버튼 선택이 안정화되었다.

## 11. `radio-test`는 되는데 `npm start`는 안 됐던 시기

한동안 가장 답답했던 구간이 바로 이거였다.

- `npm run radio-test`는 성공
- `npm start`는 실패

이건 아주 중요한 힌트를 줬다.

문제는 라디오 클릭 함수 자체가 아니라, `start`에서 그 함수가 호출되기 직전 상태였다.

실제 차이는 다음과 같았다.

- `radio-test`는 사람이 이미 띄워둔 안정된 팝업에서 시작
- `start`는 날짜 칸을 누른 직후 막 열린 팝업을 바로 조작

그래서 한 단계씩 맞춰 나갔다.

- daily mode 재클릭 제거
- 팝업 open signal 이후 바로 누르지 않기
- `결석` 옵션 4개가 렌더링될 때까지 대기

이걸 거친 뒤 `start`도 드디어 라디오 버튼을 제대로 누르게 됐다.

## 12. 저장 흐름에서 두 종류의 팝업을 만남

라디오와 사유 입력이 되자, 저장 쪽에서 또 다른 현실이 드러났다.

저장은 한 번에 끝나지 않았다.

실제 흐름은:

1. 팝업 `저장`
2. `해당자료를 저장하시겠습니까?` 확인 팝업
3. 원래 팝업 닫힘
4. `저장했습니다.` 알림 팝업

초기엔 이 중 일부를 고려하지 않아 다음 항목으로 못 넘어가거나, 실패 상태에서 넘어가려는 문제가 있었다.

그래서 저장 흐름을 “버튼 하나”가 아니라 “여러 단계의 상태 전이”로 다시 정의했다.

이 구간을 처리하기 위해:

- `confirm-test`
- robust save confirmation handler
- informational alert handler

가 추가됐다.

## 13. confirm-test를 만든 이유

저장 확인 팝업도 라디오와 마찬가지로 전체 흐름 안에서는 원인이 섞여 보였다.

그래서 이 구간도 따로 떼어:

- 이미 열린 확인 팝업에서
- `확인` 버튼만 눌러보는 모드

를 만들었다.

이 모드로 인해:

- 저장 확인 팝업 자체는 따로 눌릴 수 있는지
- 실제 버튼 좌표 클릭이 필요한지

를 분리해서 확인할 수 있었다.

이 역시 전체 흐름을 안정화하는 데 큰 도움이 되었다.

## 14. 연속출결 입력에서 발생한 실제 데이터 오류

가장 위험했던 문제는 연속출결 쪽에서 나왔다.

하루짜리는 잘 되는데:

- 시작일은 들어간 것처럼 보였고
- 종료일은 기본값이 남아서
- 의도한 `3월 6일` 대신 `3월 30일` 같은 값으로 저장되는 문제가 생겼다.

이건 단순 UI 실패가 아니라 실제 데이터 오류였기 때문에 중요도가 높았다.

원인을 다시 보면:

- 필드가 마스크 입력 컴포넌트일 가능성
- `fill()`이 내부 상태를 완전히 갱신하지 못함
- 기본값이 남아 있었음

그래서 전략을 바꿨다.

- 클릭
- `Ctrl+A`
- `Delete`
- 직접 타이핑
- `Tab`
- 실제 값 검증

그리고 입력된 값이 기대값과 다르면 저장 전에 멈추도록 만들었다.

이 시점부터는 “잘못 저장될 가능성”보다 “안전하게 실패할 가능성”이 높아졌다.

## 15. 문서화도 동시에 진행됨

중간중간 안정화된 지점마다 따로 문서를 남겼다.

먼저 남긴 것:

- 학생/날짜 타기팅 전략
- 팝업 저장 흐름

그리고 이후에는 전체 구현 설명서도 별도로 만들었다.

마지막으로 이 문서는 그 위에 올라가는 “프로젝트 여정 기록” 역할을 한다.

즉 문서 구조는 다음처럼 나뉜다.

- `attendance-targeting-strategy.md`: 학생/날짜 찾기
- `attendance-dialog-save-flow.md`: 팝업 저장 흐름
- `nice-attendance-automation-implementation.md`: 전체 기술 구조
- `project-journey-retrospective.md`: 처음부터 여기까지의 여정

## 16. 여기까지 오면서 확인된 중요한 운영 원칙

이번 작업을 통해 실제로 확인된 원칙은 다음과 같다.

1. 로그인 자동화보다 세션 attach가 훨씬 현실적이다.
2. 나이스 같은 업무 시스템은 동적 ID보다 화면 신호와 상태 검증이 더 중요하다.
3. 가상 스크롤이 있는 화면은 학생 이름 기준으로 접근해야 한다.
4. 팝업 조작은 일반 selector click보다 “활성 dialog 한정 + 좌표 클릭 + 상태 확인”이 더 강하다.
5. 보조 모드를 만드는 것이 전체 자동화 디버깅보다 훨씬 빠르다.
6. 날짜 입력은 실제 입력값 검증까지 해야 한다.
7. 저장은 하나의 액션이 아니라 여러 확인 단계를 포함하는 흐름이다.

## 17. 현재 시점의 성과

현재까지 가능한 것:

- 로그인된 브라우저 세션 연결
- 나이스 창 선택
- 출결관리 화면 기준 자동화
- 스크롤 상태와 무관한 학생/날짜 칸 탐색
- 하루짜리 출결 입력
- 연속출결 입력
- 질병/미인정/기타/출석인정 선택
- 사유 입력
- 저장 확인 처리
- 저장 완료 알림 처리
- 종료일 오입력 방지

즉, 처음의 “셀렉터가 자꾸 바뀌는 것 같다”는 불안정한 출발점에서, 실제 업무 입력이 되는 수준까지 도달했다.

## 18. 다음 단계

현재 다음 큰 작업은 `비고등록` 자동화다.

이건 기존 흐름 위에 바로 얹기보다:

- 학생/날짜 타기팅 로직은 재사용
- 비고등록 버튼/팝업/저장 흐름만 별도 단위로 구현

하는 편이 좋다.

지금까지 작업 경험상, 이 방식이 가장 안전하고 유지보수도 쉽다.

## 마무리

이 프로젝트는 처음부터 “한 번에 되는 자동화”가 아니었다.

대신:
